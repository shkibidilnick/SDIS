# Архитектура проекта

Этот документ описывает архитектурные решения, принятые при разработке Super Mario Bros для лабораторной работы №3.

## Цели проектирования

1. **Тестируемость** — основная игровая логика (физика, рекорды, конфиги, поведение сущностей) должна быть покрыта юнит-тестами без необходимости запускать pygame.
2. **Расширяемость** — добавить новый тип врага, бонуса или поменять баланс игры должно быть возможно правкой JSON и максимум одной фабрики.
3. **Прозрачность зависимостей** — pygame используется только в слоях ввода-вывода (рендер, звук, события), но не в логике.
4. **Соблюдение SOLID, DRY, KISS, YAGNI.**

## Ключевые классы

### `PhysicsEntity` — база сущностей

Содержит общий код, нужный игроку, врагу и бонусу:

- хранение `Rect` положения,
- скорости `vx`, `vy`, флаги `on_ground`, `alive`,
- метод `apply_gravity(g, max_fall)` — гравитация с ограничением,
- метод `move_and_collide(level)` — перемещение с раздельной обработкой горизонтального и вертикального движения и коррекцией позиции при столкновении с твёрдыми тайлами. Возвращает словарь с информацией о направлениях столкновения.

Это **единственное место**, где реализована физика. Player/Enemy/PowerUp вызывают её, а сами реализуют свои поведенческие надстройки.

### `Level` — данные карты

Хранит сетку тайлов как список строк (по символу на тайл). Предоставляет:

- `get_tile(tx, ty)` / `set_tile(...)` — чтение/запись;
- `is_solid(tx, ty)` — проверка коллизии;
- `solid_tiles_in_rect(rect)` — все твёрдые тайлы, пересекающие прямоугольник;
- `interactive_tiles_in_rect(rect)` — для подбора монет и достижения флага.

Карта валидируется при создании: высота и длины строк должны соответствовать заявленным размерам.

### `Player`

Имеет два состояния: `small` и `big`. Применённый гриб переводит в `big`; получение урона от врага в `big` возвращает в `small` и даёт временную неуязвимость. В `small` урон = смерть и потеря жизни.

#### Эффекты бонусов

- **гриб** (`grow`) — переводит в `big`, действует **20 секунд** (1200 кадров), потом возврат в small с короткой неуязвимостью;
- **звезда** (`invincible`) — таймер `star_timer` на **10 секунд** (600 кадров);
- **огненный цветок** (`speed_boost`) — таймер `speed_boost_timer` на **12 секунд** (720 кадров).

#### UX-улучшения управления

- **Jump buffer** (`jump_buffer_timer`, 6 кадров): если игрок нажал прыжок до приземления — прыжок выполнится при касании земли.
- **Coyote time** (`coyote_timer`, 6 кадров): после схода с платформы прыжок ещё разрешён несколько кадров.

`update()` возвращает структуру с информацией о коллизиях, чтобы `GameScene` могла отреагировать (выбить блок `?`, разрушить кирпич, забрать монету).

### `Enemy` и поведения

Тип врага задаётся полем `behavior`:

- **`walk`** (Гумба) — идёт в текущем направлении, разворачивается при столкновении со стеной и **перед обрывом** (для последовательного поведения на парящих платформах).
- **`walk_smart`** (Купа) — то же самое плюс заранее проверяет тайл под ногами впереди.
- **`piranha`** (Пиранья) — особое поведение без гравитации и коллизий. Цикл состояний:
  - `hidden` — спрятана внутри трубы (90 кадров);
  - `emerging` — выезжает вверх (анимация подъёма);
  - `shown` — полностью видна (120 кадров);
  - `retracting` — уходит обратно вниз;
  - снова `hidden` (120 кадров) — и цикл повторяется.

Отрисовка пираньи использует `clip` pygame, чтобы спрайт обрезался по верхней границе трубы — так она визуально «уходит» в трубу, а не просто исчезает.

#### Реакции на убийство

- `stomp()` — игрок прыгнул сверху (только для `can_be_stomped: true`);
- `kill_by_star()` — звезда / неуязвимый Марио, или большой Марио для Гумбы.

#### Случайная вариация скорости

В `__init__` базовая скорость умножается на `random.uniform(0.85, 1.15)` — лёгкое разнообразие в поведении.

### `PowerUp` и фабрика

Появление из `?`-блока — анимация подъёма на одну клетку (`EMERGE_FRAMES = 30`). После завершения эмерджа применяется обычная физика. Звезда дополнительно подпрыгивает при касании земли.

Содержимое `?`-блока выбирается **случайно** при ударе с весами:
- гриб — 40%;
- огненный цветок — 35%;
- звезда — 25%.

### Сцены

Все наследуются от `Scene`:

```python
class Scene:
    def __init__(self, app): ...
    def handle_events(self, events) -> None: ...
    def update(self, dt) -> None: ...
    def draw(self, surface) -> None: ...
    # done = True + next_scene = ... — переход
```

`App` вызывает методы текущей сцены в цикле, а при `done=True` переключается на `next_scene`.

### Утилиты

- **`ConfigLoader`** — статические методы загрузки JSON, кеш для повторных запросов.
- **`Rect`** — собственная реализация AABB без pygame (для тестируемости).
- **`HighScoreManager`** — таблица топ-5; нормализация имени (uppercase, ≤10 символов, пустое → `"ANON"`). Метод `is_high_score` корректно проверяет попадание в топ-5 при любой степени заполнения таблицы.
- **`AudioManager`** — генерирует WAV синтезом (синус/квадрат/пила + огибающая), кеширует на диск. При недоступном миксере все методы становятся no-op.
- **`Renderer`** — рисует игрока, врагов (включая пиранью с clipping), бонусы, тайлы (с расширенной крышкой трубы), частицы и UI. Все спрайты — программные.

## Расширяемость

### Добавление нового типа врага

1. В `enemies.json`, секция `enemies`, добавить запись с уникальным `behavior`.
2. В `Enemy.update()` (или в одном из специализированных методов вроде `_update_piranha`) добавить ветку для нового `behavior`.
3. (Опционально) Добавить отрисовку в `Renderer.draw_enemy`.

### Добавление нового типа бонуса

1. В `powerups.json` описать в `powerups`.
2. В `Player.apply_powerup()` добавить ветку обработки нового `effect`.
3. В `GameScene._handle_block_hit` добавить новый тип в список `random.choices`.

### Изменение карты

Полностью декларативно через `level_map.json`. Используемые символы:
- `.` — пусто
- `G` — земля (твёрдая)
- `B` — кирпич (твёрдый, разрушается big-Марио)
- `Q` — `?`-блок (твёрдый, выбивается снизу)
- `P` — труба (твёрдая)
- `C` — монета (интерактивная)
- `F` — флаг (интерактивная, конец уровня)
- `S` — ступенька

## Решения, заслуживающие внимания

### Почему свой `Rect`, а не `pygame.Rect`?

`pygame.Rect` использует целочисленные координаты, что плохо подходит для физики (теряется sub-pixel точность скоростей вроде `0.6`). Кроме того, импорт pygame сделал бы невозможным юнит-тестирование без графики. Собственный `Rect` решает обе проблемы.

### Почему звуки и спрайты генерируются, а не лежат как файлы?

- **Спрайты** — рисуются примитивами в `Renderer`. Это упрощает репозиторий (нет бинарных ассетов), убирает вопросы лицензирования и делает оформление настраиваемым через цвета в JSON.
- **Звуки** — синтезируются в `AudioManager` (короткие синус/квадрат/пила-волны с огибающей), кешируются в `assets/sounds/*.wav` после первого запуска. Музыка — простая мелодия из 16 нот в C-major.

### Почему `move_and_collide` обрабатывает X и Y раздельно?

Это стандартный приём для AABB-коллизий: если двигать сразу по обеим осям, нельзя надёжно определить, по какой стороне произошло столкновение. Раздельная обработка корректно обрабатывает «скольжение» вдоль стены и «приземление» одновременно с боковым движением.

### Почему пиранья — отдельная ветка `_update_piranha` без гравитации?

Пиранья концептуально отличается от ходячих врагов: она не подчиняется гравитации, не имеет AABB-коллизий с тайлами, и её движение целиком определяется конечным автоматом из 4 состояний. Помещать её в общий `update` означало бы добавлять флаги вроде `ignore_gravity`, что усложнило бы базовый класс. Раздельная ветка соблюдает SRP и оставляет код Гумбы/Купы простым.

### Почему `_handle_block_hit` использует `random.choices` с весами?
